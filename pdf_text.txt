Section 01
Executive Summary
EYES is a personal digital memory operating system. It indexes a user's full online history through
OAuth connectors, makes that history queryable through an AI interface with citations, and detects
commitments and opportunities in incoming data streams in real time.
V1 is one complete consumer web application. A user signs up, connects two or more accounts via
OAuth, the system indexes their historical data, and the user can query that data with citations. Inside
the same application sits the Reputation Audit, a paid feature priced at €39, which generates a
downloadable PDF report. Real-time commitment detection runs on incoming webhooks. The
application also exposes itself as an MCP server so it can be queried from Claude Desktop, Cursor, and
any other MCP-compatible AI client.
What Crescent Moon delivers
By end of sprint, the Crescent Moon team must deliver, in order of priority:
01
Complete migration of all code, infrastructure, accounts, and intellectual property to
Abhi-owned identity.
02
A working V1 web application: OAuth connection flow, indexed memory, query interface with
citations, MCP server exposure.
03
Reputation Audit feature: live preview in-app and full PDF export, branded, watermarked,
GDPR-compliant.
04
Real-time commitment detection writing to a dedicated database table.
05
Mistral AI abstraction layer in code from day one (no Mistral integration required, only the
abstraction).
06
Behavioral data capture (Layer 2) instrumented from the first user.
07
Documentation, deployment guides, and a recorded acceptance test.
What Crescent Moon does not retain
Upon delivery, Crescent Moon retains no rights, no access, no credentials, and no intellectual property
associated with EYES. All copies of code, schemas, prompts, training data, design assets, and
credentials in their possession must be deleted, with written confirmation provided to Abhi within seven
days of final sign-off. This is not negotiable and is consistent with the IP transfer agreement already in
place.

Section 04
V1 Product Specification
V1 is one consumer web application. It is not a prototype. It is not a demo stub. It is the
product Abhi will sell to first paying customers and demonstrate live to Y Combinator partners.
It must work without rehearsal.
Core capabilities
01. User authentication and onboarding
A user can sign up using email and password. Magic-link or passwordless authentication is acceptable
as an alternative. After signup, the user is directed to the onboarding flow, which requires connecting at
least one OAuth source before the application becomes usable. Onboarding includes the Privacy
Shields configuration step and the Sync Depth selection step.
02. OAuth connector framework
The user can connect any of the supported platforms through a single, uniform UI. For V1, Gmail and
Reddit are mandatory. Beyond those two, any of the existing six (Slack, GitHub, Discord, Notion,
Google Calendar) should remain working as they do today, plus any of the additional six (X, Trello,
Netlify, Canva, PostHog, Dropbox) that can be brought to a production-ready state during this sprint.
Connectors that do not reach production-ready state by sprint end must be hidden from the UI rather
than shipped broken.
03. Indexing pipeline
When a connector is added, the application begins indexing the user's historical data on that platform.
Indexing must complete within five minutes per connector under typical conditions (a Gmail account with
up to 50,000 messages over ten years; a Reddit account with up to 5,000 comments and posts). Where
the data volume exceeds these thresholds, the system indexes in chronological priority order (most
recent first) so that queries on recent data work even before full indexing completes. The user is shown
indexing progress in real time, per platform.
If indexing exceeds five minutes for the typical case described above, the application is not
ready for handover. This is a hard requirement.
04. Query interface
The user types a natural language query into a single prominent input. The system retrieves relevant
indexed records using vector similarity search (via pgvector) augmented with keyword filters. The
retrieved records are passed to Claude with a system prompt that instructs Claude to answer the query
using only the provided records, and to cite the source of each factual claim by referencing the record
ID. The UI then renders the answer with inline citation chips that link to the original source (e.g. the
exact email, Reddit comment, or calendar event). Citation links must actually open the source. Broken
citations are unacceptable.

06. Reputation Audit
See Section 05 for the complete specification. In summary: the user pays €39 via Stripe, the system
runs the audit pipeline on their indexed data, and produces both an in-app preview (revealing only
summary signals) and a downloadable PDF (containing the full report including raw citations). This is
the demo-critical feature. It must work flawlessly in front of Y Combinator partners.
07. Commitment detection
When new data arrives via Gmail Pub/Sub webhooks (or polling, for sources without push notifications),
each new record is passed through a Claude call with a tight system prompt asking: does this record
contain a commitment, deadline, or opportunity? If yes, extract the commitment text, the likely due date
if mentioned, and the type. Insert into the detected_commitments table. The Claude call must complete
within two seconds; if it does not, the prompt must be tightened until it does. For V1, no proactive alerts
are required. Detections are stored and displayed in the Commitments view.
08. MCP server
EYES exposes itself as an MCP server. Other AI clients (Claude Desktop, Cursor, OpenCode, etc.) can
connect to a user's EYES instance using the user's API key, and call tools such as query_my_history,
get_recent_commitments, and find_communications_about. The MCP server implementation must
follow the Anthropic MCP specification. Authentication is per-user via API key generated in the user's
account settings.
09. Privacy Shields
The user can specify a list of email senders, domains, or keywords that are excluded from indexing.
This list is enforced before any data enters the raw_events table. Records matching an exclusion are
not stored, not embedded, not indexed, not queryable. The exclusion list is applied to the initial backfill
and to all incoming webhook events. The user can modify the list at any time; modifications take effect
on future events only. To purge already-indexed data matching a new exclusion, the user must explicitly
trigger a re-index.
10. Sync Depth
The user can choose Shallow (last 30 days), Balanced (last 6 months), or Deep (full available history).
This setting affects the initial backfill. Once set, it can be changed; changing from a shallower to a
deeper setting triggers an incremental backfill of the additional historical window.
11. Risk Sensitivity
The Neural Tuning slider controls how aggressively the commitment detection model flags potential
risks, opportunities, and overdue commitments. At low sensitivity, only explicit deadlines are flagged. At
high sensitivity, implied commitments and soft promises are also flagged. The slider value (0–100) is
passed into the Claude system prompt for the commitment detection step.

Section 05
Reputation Audit Specification
The Reputation Audit is the demo-critical feature. It is what Y Combinator partners will see live.
It is what the first paying customer pays €39 for. The entire investment thesis of EYES — that
personal data is an asset the owner should be able to inspect, not a resource that platforms
exploit — is communicated through this single feature. It must be exceptional.
Tone and feel
The audit is cold, clinical, and direct. It does not flatter the user. It does not soften findings. It is
structured like an intelligence report produced by Palantir, written with the editorial discipline of an
Economist leader column. Where competitors would say "It looks like you might want to consider
following up with...", the audit says "You committed on April 22 and have not followed up. The deadline
passed three days ago."
The audit's purpose is to find the things the user has missed. Every page of the report should
reinforce that mission.
Scope
The audit is dynamic. It runs against whatever connectors the user has active at the moment of audit
purchase. If the user has only Gmail and Reddit connected, the audit covers Gmail and Reddit. If the
user has all six live connectors, it covers all six. The PDF report adapts its section count to the
connectors covered.
The audit operates only on the user's authorized connector data. It does not search the public web. It
does not query third-party data brokers. It does not enrich with external information. This must be stated
explicitly in the report's GDPR notice.
Two surfaces: live preview and PDF export
The audit produces two distinct experiences from the same underlying analysis run.
Live preview (in-app)
Shown immediately after audit completes, inside the EYES web application. The preview reveals only
the summary signals: the risk score, the count of mentions per platform, the count of detected
commitments, and a single headline finding ("You have 3 unfulfilled commitments referenced in
investor-facing communications"). The full citations, raw quotes, and platform breakdowns are not
shown in the preview. A prominent "Download full report" button triggers the PDF export.
Why this split: the live preview is what is shown during the demo (intriguing, premium, controlled). The PDF is what
the customer keeps and uses (comprehensive, detailed, citable). Both must be designed to brand.
PDF export
Generated server-side using ReportLab (or equivalent). Targeted length: 6 to 8 pages of A4.
Single-column layout. Cream background (#FAFAF7). Ink black body text on cream. Forest green

PDF report structure
The PDF follows this fixed structure. Section count adapts to active connectors but the order and styling
do not vary.

EYES wordmark, title "Reputation Audit Certificate", subject's name (as authenticated), date and time generated,

One paragraph plain-language narrative summary. Below it, three headline metrics in a row: total mentions

One page per connector covered. Each page shows: the platform name, indexing window, total records analysed,

The findings the user is most likely to be unaware of: contradictions between public-facing statements, stale

decides what to do about it.

Full enumerated list of every source citation referenced in the report. Each entry includes the platform, date,
identifier, and a short excerpt. Below the citations index, the GDPR notice, the data-source disclosure, and the
EYES legal footer.

Audit generation pipeline
The pipeline runs server-side. End-to-end completion target: under 60 seconds for typical accounts. If
generation will take longer, the user is shown a progress screen and the PDF is emailed when ready.
Step 1. User completes Stripe checkout. Webhook fires audit.purchase.completed event. Server-side
handler creates an audit record in the database with status pending.
Step 2. Pipeline pulls the user's raw_events for the last two years (configurable). For each connected
platform, fetches up to 5,000 most recent records.
Step 3. Runs Claude with extraction prompt: "Find every mention of [user's name] and any associated
identifiers. For each mention, extract: source, date, sentiment (+1, 0, -1), is this a commitment made by
the subject, is this a commitment made to the subject, is this potentially sensitive."
Step 4. Aggregates extraction output. Computes risk score: (negative_count × 2 + neutral_count × 0.5 +
unfulfilled_commitment_count × 3) / total_mentions × 10, capped at 10.
Step 5. Runs second Claude call to generate executive summary narrative. Cold tone, no flattery.
Step 6. Renders PDF using ReportLab template. Stores PDF in Supabase storage bucket under
audits/{user_id}/{audit_id}.pdf with appropriate access policy.
Step 7. Returns signed download URL to client. Sends email to user with the same URL. Updates audit
record status to completed.
Risk score methodology (visible in report)
The methodology is published in the report itself, in a small typeset block on the executive summary
page. Transparency about the score is part of the brand promise.
Risk score = ( ( count(negative_mentions) × 2 ) + ( count(neutral_mentions) × 0.5 ) +
( count(unfulfilled_commitments) × 3 ) ) ÷ count(total_mentions) × 10, capped at 10.
Recency weighting is applied to the underlying counts: mentions in the last 30 days carry weight 1.0,
last 6 months carry 0.5, older than 6 months carry 0.2. This is also stated in the methodology block.

Section 07
Mistral AI Integration & Behavioral Data Layer
EYES will not call Mistral AI in V1. Every model invocation in V1 uses Claude (for reasoning,
citation, and detection) and OpenAI (for embeddings). However, two architectural decisions
must be made now to preserve the path to a Mistral fine-tune around month seven, when the
behavioral dataset becomes large enough to justify training. Failure to instrument these now
requires invasive refactoring later.
Decision 1: Model abstraction layer
Every model call in the codebase must go through a single abstraction rather than directly invoking the
Anthropic or OpenAI SDK. The abstraction supports model selection by capability (chat, embed,
classify) rather than by provider, and selects the underlying provider based on configuration.
Indicative interface (final form decided by the engineering team, but must satisfy this contract):
from eyes.models import invoke_model
result = invoke_model(
    capability="chat", # chat | embed | classify
    system=system_prompt,
    messages=messages,
    preference="claude", # claude | mistral | openai | auto
    capture=True, # log to behavioral data layer
)
When preference is "auto" and the configuration enables it, the layer may route the call to a fine-tuned
Mistral endpoint for the categories of queries on which the fine-tune outperforms (or matches) Claude.
In V1 the Mistral configuration is empty, so "auto" always resolves to Claude. The presence of the
routing layer means flipping that switch in V2 requires zero application changes.
Decision 2: Behavioral data capture (Layer 2)
From the first user, EYES captures anonymized behavioral data about how users query their own past.
This dataset is the foundation of the Mistral fine-tune (month 7) and the long-term moat of EYES. A new
table is added to the schema:
CREATE TABLE query_behavior (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_hash TEXT NOT NULL, -- SHA-256 of user_id + salt
    query_text TEXT NOT NULL,
    query_type TEXT NOT NULL, -- audit | search | commitment | mcp
    sources_used TEXT[] NOT NULL, -- which connectors fed retrieval
    result_count INT NOT NULL,
    response_length INT NOT NULL,
    model_used TEXT NOT NULL, -- claude | mistral | openai
    latency_ms INT NOT NULL,
    quality_signal NUMERIC, -- placeholder for future feedback
    coarse_geography TEXT, -- country code only, never IP
    coarse_time_bucket TEXT, -- morning | afternoon | evening | night
    created_at TIMESTAMPTZ DEFAULT now()

Anonymization rules. user_hash is computed as SHA-256(user_id || global_salt) where global_salt is a

Stack consistency

Code organization
The codebase is organized so that someone joining the project can locate any feature within five
minutes. Folder structure follows feature boundaries: connectors, audit, query, mcp_server,

database queries. Shared utilities live in a lib folder. Tests live alongside the code they cover.
Documentation
The repository contains, at minimum: a README explaining how to run the project locally; an

explaining how the production deployment is configured; an OAUTH document listing every OAuth

wired, otherwise Vercel logs are acceptable for V1). Errors are captured with stack traces and user
context. Slow query alerts fire if any query exceeds the performance threshold defined in Section 06.
Security baseline
Row-level security policies on every Supabase table containing user data. OAuth tokens encrypted at
rest using Supabase's vault feature or equivalent. Service-role key never exposed to the client. All
server-to-server calls authenticated. Stripe webhooks verified against Stripe signing secret. CORS
policy restrictive — only the production domain is allowed.
