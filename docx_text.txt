The original spec you worked from positioned EYES as a search engine for your own digital history — a tool that lets you query everything you've ever said. That product is real. You built it. It works. Six OAuth connectors live, vector retrieval functional, cited answers shipping.
That product, as it stands today, is what every other personal AI memory company is also trying to build. Mem0, Supermemory, Vellum, Hark, the abandoned Rewind AI — they are all in the search-and-retrieve category. If we ship only that, we are one of seven companies competing on the same axis, and we lose because they are bigger, better-funded, and earlier.
The product has evolved. EYES is now a personal intelligence layer — a system that models humans across time and surfaces patterns they cannot see from inside their own life. That is a different category. That category has zero credible entrants today. Building it is what justifies the company existing.
ONE-LINE DEFINITION — EYES = a system that sees you across time better than you see yourself. Not a search engine. Not a chatbot with memory. A behavioral intelligence layer that reveals patterns, loops, drift, and probable trajectories using your own digital history as evidence.
Concretely: the substrate you built is the floor of the new product, not the whole product. Six OAuth connectors, pgvector, cited retrieval — those become inputs to a cognitive layer that did not exist in the original spec. The cognitive layer is what you are building in the next 5 weeks.
Specifically, V1 now contains five new capabilities that did not exist in your previous spec:
State clustering — the system learns that each user cycles through 6–10 distinct cognitive modes (e.g., "expansion mode," "isolation cluster," "investor-prep") and can tell them which one they're in right now, today.
Loop detection — recognizes when the user is running a behavioral pattern they've run before ("this is the 4th time you've entered isolation-mode after an investor rejection — last 3 times it lasted 18–22 days").
Drift detection — compares what the user says they value (Notion docs, voice memos, journal entries) against what they actually do (calendar, message volume, output) and surfaces the gap.
Entity correlations — maps which people, places, apps, and times shift the user into which cognitive states ("conversations with [name] correlate +0.78 with expansion-mode").
Forward inference — gives the user probability-weighted estimates of their likely next state, always cited against their own history.
These five capabilities, layered on top of the substrate you already built, are V1. That is what we are shipping by June 13, 2026.

1.3 — The dual-layer architecture (the single most important concept)
EYES has always been one product, but underneath the single chat interface there are two simultaneously running systems. They are not features. They are not modes. They are not options the user toggles between. They both run, all the time, on the same data substrate.
If you understand this section, everything else in this document follows naturally. If you don't, you will build the wrong thing. Read it twice.
System 1 — The acute layer (directed actions)
Watches every incoming data point in real time. When something arrives that contains an explicit demand, commitment, deadline, or ask, the acute layer reads it within seconds and acts on it.
Example: Valentin emails you on Wednesday asking about the deck you said you'd send. While that email is still landing in your inbox, the acute layer has already:
Read the email and parsed it as an ask.
Searched the user's history for any commitment matching "deck" + "Valentin".
Found the original promise from 3 weeks ago: "I'll send you the deck Thursday."
Surfaced the whole thing as one notification: "Valentin is asking about the deck. You promised it on April 17. Today is May 8."
This layer is supervised. The label is in the data. The demand is in the words. It is deterministic. When it fires, the user trusts it because the citation chain is bulletproof.
Most of this layer is already built. You have webhooks, you have retrieval, you have citation. The remaining work is the detection pass — the second Claude call that classifies each incoming event.
System 2 — The chronic layer (pattern detection)
Runs in the background, continuously, on the accumulated history. Aggregates each day's behavior into a single high-dimensional state vector. Clusters those vectors over time. Detects recurrence. Surfaces drift. Reveals entity correlations. Estimates probable trajectories.
This layer is unsupervised. There are no labels. The system has to find structure on its own. To make sure the structure it finds is real and not hallucinated, the chronic layer is gated by three constraints:
Bounded data. The system is not asking "what patterns exist in human behavior?" — it's asking "what patterns recur in this specific user's data?" That question is finite and computable.
Statistical gating. Clusters and loops only surface if the math says they are real. HDBSCAN labels low-confidence points as noise. Recurrence analysis only fires if periodicity is statistically significant. Silence is a valid output.
User validation. Detected clusters are shown to the user as drafts. The user names them. The user confirms or rejects. Only after confirmation does a cluster label become operational.
This layer is what you are building. It is new. It is the entire purpose of the next 5 weeks.
How the two layers work together
They are not parallel products. They feed each other.
Acute fires → an email lands, the system flags an ask connected to a prior commitment.
Chronic adds context → the user is currently in their isolation-mode cluster. The system says: "You're being asked about the deck. Note: you're in the cluster where you historically over-promise on deadlines. The last 3 commitments you made in this state were reversed within 2 weeks. Want to draft a reply now or sleep on it?"
Acute without chronic is a smart notification system. Chronic without acute is an unused analytics dashboard. Together they are the product.
TIMELINE — Acute carries days 1–30. The user feels the product immediately because the acute layer fires from day one. Chronic comes online progressively — it needs ~30 days of state vectors to find stable clusters. We are building both. They run on the same substrate. They surface through the same chat interface.

2.2 — The ingestion pipeline (already built — small additions)
You already have an ingestion pipeline. It pulls data from OAuth connectors, sanitizes it, generates embeddings, and stores it in pgvector. This works.
Three small additions for V1:
Addition 1 — Tag every ingested item with date_bucket
Every item we ingest should have a date_bucket field (YYYY-MM-DD) representing the day it occurred. This is what the chronic layer uses to aggregate daily state vectors. Add this column to your existing memories table (or whatever you call it).
-- Migration
ALTER TABLE memories ADD COLUMN date_bucket DATE;
CREATE INDEX idx_memories_user_date ON memories(user_id, date_bucket);
 
-- Backfill
UPDATE memories SET date_bucket = DATE(created_at) WHERE date_bucket IS NULL;
Addition 2 — Tag every item with content_type
We need to distinguish between "stated" content (Notion docs, voice memos, journal entries, manifesto-style messages) and "lived" content (calendar events, code commits, message counts, time-of-day signals). Drift detection depends on this split.
Add a content_type column with these values:
stated — Notion pages, voice memo transcripts, long-form Slack messages (>200 chars), email drafts you sent (not received), tweets/posts.
lived — calendar events, GitHub commits, Slack message timestamps (without content), email receive-times, short reaction messages.
mixed — anything that doesn't cleanly fit (default for emails received, where content matters but so does timing).
Classification logic is simple. Run it at ingestion time:
function classifyContent(item) {
  if (item.source === 'notion') return 'stated';
  if (item.source === 'voice_memo') return 'stated';
  if (item.source === 'calendar') return 'lived';
  if (item.source === 'github_commit') return 'lived';
  if (item.source === 'slack' && item.length > 200 && item.is_outbound) return 'stated';
  if (item.source === 'gmail' && item.is_outbound && item.length > 300) return 'stated';
  return 'mixed';
}
Addition 3 — Tag every item with entities[]
Entity correlations depend on knowing which people, places, and tools are mentioned in each item. Use a simple Claude call at ingestion time to extract them. Schema:
{
  entities: [
    { type: 'person', name: 'Valentin Henry', canonical_id: 'person_001' },
    { type: 'organization', name: 'Innostart', canonical_id: 'org_004' },
    { type: 'tool', name: 'Notion', canonical_id: 'tool_002' }
  ]
}
Store this as JSONB in a new entities_extracted column. Canonical IDs come from a per-user entities table — first time "Valentin" appears, create a new entity row. After that, match by name + context using a simple Claude prompt.
COST NOTE — Don't over-engineer entity extraction. Use Claude. It's expensive per call but you're indexing maybe 500 items per user per day. That's ~$0.15/user/day in entity extraction. Acceptable for beta.
2.3 — The acute layer (real-time event detection)
The acute layer is what fires on incoming data. It is the "Valentin email" experience from §2.1. Most of this is already built — webhooks, ingestion, retrieval. What's missing is the detection pass.
Architecture
Incoming event (Gmail webhook, Slack event, Calendar push, GitHub event)
         |
         v
  [Ingestion] — write to memories table
         |
         v
  [Detection pass] — Claude call: "is this an ask, commitment, deadline, or reference to prior commitment?"
         |
         v
  If yes: [Cross-reference] — search memories for related commitments
         |
         v
  If match found: [Surface] — push notification to chat interface
The detection prompt
Every incoming event runs through this Claude call. Use claude-3-5-haiku for cost (this fires hundreds of times per user per day):
System prompt:
You are EYES, a personal intelligence layer. You classify incoming events.
Given an event, output ONE of these classifications as JSON:
 
{
  classification: 'ask' | 'commitment' | 'deadline' | 'reference' | 'noise',
  confidence: 0.0-1.0,
  extracted: {
    entities: [...],
    topic: string,
    deadline_date: string | null,
    requires_response: boolean
  }
}
 
Definitions:
- 'ask': someone is requesting something from the user (info, decision, action)
- 'commitment': user is making a promise (drafted by them, going outbound)
- 'deadline': time-bound obligation
- 'reference': someone is referencing a prior conversation/commitment
- 'noise': none of the above (newsletter, ad, automated, etc.)
 
Be conservative. When unsure, classify as 'noise'.
The cross-reference logic
If detection returns ask, commitment, or reference with confidence > 0.7, run a vector search against the user's history. Find the top 3 related items. Pass them + the new event to Claude for synthesis.
Synthesis prompt:
Given this new event and these related prior events, determine:
1. Is there a connection? (Y/N)
2. If yes, what's the connection in one sentence?
3. Should we surface this to the user? (Y/N)
4. If yes, draft the surfacing message in EYES tone (direct, accountable, no fluff).
 
Output: {
  connected: boolean,
  connection_summary: string,
  should_surface: boolean,
  surface_message: string,
  citations: [memory_id, memory_id, ...]
}
Surfacing
If should_surface = true, write a row to the alerts table. The chat interface polls this table every 30 seconds and shows new alerts at the top of the conversation.
SCOPE — Do not push notifications to email or browser in V1. Surface only inside the chat UI. The user opens EYES and sees the alert. Push notifications are a Month 2 feature once we know what users actually care about.
Acceptance criteria
New Gmail message arrives → webhook fires → detection runs within 10 seconds.
If ask/commitment/reference detected → cross-reference completes within 20 seconds.
If surface triggered → alert appears in chat UI within 30 seconds of original event.
Citations in alert link to actual source emails/messages.
False positive rate < 20% (user dismisses ≤ 1 in 5 alerts as noise).

export async function GET(req: NextRequest) {

Cron runs at 3am UTC daily without manual intervention.
Every active user has exactly one state_vectors row per day (including days they were inactive — vector for inactive day is the zero-vector with metadata.is_inactive = true).
Per-user computation completes in under 5 seconds.
Failures are logged with user_id and don't block other users from being processed.
2.5 — State clustering (the cognitive core)
This is the section that turns EYES from a search engine into an intelligence layer. Read it carefully. Implement it carefully.
What we're doing
Once a user has ~21 days of state vectors, we run dimensionality reduction (UMAP) + density-based clustering (HDBSCAN) on their trajectory through state-space. The output is between 3 and 10 stable clusters representing their distinct cognitive modes.
We then ask Claude to describe each cluster in human-readable terms. We then ask the user to confirm and name each cluster. After confirmation, cluster labels become operational throughout the chat interface.
The algorithm
Step 1 — Pull last 90 days of state vectors for the user (or all available if < 90).
Step 2 — Stack them into an N x 1556 matrix.
Step 3 — Run UMAP to reduce to N x 8 (good clustering dimensionality).
Step 4 — Run HDBSCAN on the reduced matrix.
Step 5 — Get cluster labels per day (some days will be labeled -1 = noise; this is correct).
Step 6 — For each cluster, gather the days that belong to it and compute a signature.
Step 7 — Ask Claude to describe each cluster.
Step 8 — Show drafts to user. User names them or rejects them.
Step 9 — Store confirmed clusters with user-provided labels.
Implementation — Python service or JS?
UMAP and HDBSCAN are Python libraries. You have two options:
Option A (recommended for V1): Deploy a tiny Python microservice on Modal or Railway. Call it from your Next.js cron. ~$5/month for our scale.
Option B: Use the JS implementations (umap-js + density-clustering). Slower, less battle-tested, but no extra infra.
DECISION — Go with Option A. The Python implementations are mature and you'll save days of debugging. Modal has a generous free tier and zero ops overhead.
Python service code
# clustering_service.py — deploy to Modal
import modal
import numpy as np
from umap import UMAP
from hdbscan import HDBSCAN
 
app = modal.App('eyes-clustering')
image = modal.Image.debian_slim().pip_install('umap-learn', 'hdbscan', 'numpy')
 
@app.function(image=image)
def cluster_user(vectors: list, min_cluster_size: int = 5):
    """
    vectors: list of lists, each of dim 1556
    Returns: list of cluster labels (ints, -1 = noise)
    """
    arr = np.array(vectors)
    if len(arr) < 21:
        return [-1] * len(arr)  # Not enough data
 
    # Reduce to 8 dims
    reducer = UMAP(n_components=8, n_neighbors=min(15, len(arr)-1), random_state=42)
    reduced = reducer.fit_transform(arr)
 
    # Cluster
    clusterer = HDBSCAN(min_cluster_size=min_cluster_size, min_samples=3)
    labels = clusterer.fit_predict(reduced)
 
    return labels.tolist()

User: Here are 5 days from a recurring state in this user's data. Each day includes message volume, dominant connectors, key topics, time-of-day pattern, and notable activity.
 
[5 days of data...]
 
Output as JSON:
{
  draft_label: "4-6 word description of this state",
  signature: "3-sentence description of what's distinctive about this state",
  representative_quote: "a quote from the user's content that captures this state",

}
The user validation moment
Once cluster descriptions are generated, surface them in the chat interface. Don't push notification, don't email — just queue a one-time modal that fires on the user's next visit to EYES.
The modal shows 3-7 cluster cards (one per detected cluster). Each card shows the draft label, signature, representative quote, and frequency. The user can:
Accept the draft label as-is.
Edit the label and accept.
Reject this cluster (mark it as not real / noise).
Only confirmed clusters get used operationally. Rejected clusters are flagged so we don't re-show them next clustering run.
Schema
CREATE TABLE clusters (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id),
  draft_label TEXT NOT NULL,
  user_label TEXT,  -- null until confirmed
  signature TEXT NOT NULL,
  representative_quote TEXT,
  status TEXT NOT NULL DEFAULT 'draft',  -- 'draft', 'confirmed', 'rejected'
  cluster_version INT NOT NULL,  -- incremented each re-cluster
  created_at TIMESTAMPTZ DEFAULT NOW(),

2.6 — Loop detection (recurrence analysis)
Once we have cluster assignments per day, we have a sequence: cluster_A, cluster_A, cluster_B, cluster_B, cluster_C, cluster_A, cluster_A, ... Loop detection is about finding statistically significant recurrence in this sequence.
The simple version (V1)
Don't reach for full recurrence quantification analysis in V1. Use a simpler approach that catches the obvious patterns:
For each cluster the user has been in, compute a list of "entries" — the start dates of each contiguous run of that cluster.
For clusters with 3+ entries, compute the mean and stddev of inter-entry intervals.
If stddev < 30% of mean (i.e., the cluster recurs with consistent periodicity), flag it as a loop.
Also compute the average duration of each run. Flag if duration is consistent (stddev < 40% of mean).
Code
function detectLoops(assignments: ClusterAssignment[]) {
  const loops = [];
  const byCluster = groupBy(assignments, 'cluster_id');
 
  for (const [clusterId, dates] of Object.entries(byCluster)) {
    const runs = findContiguousRuns(dates);  // [{start, end, duration_days}, ...]
    if (runs.length < 3) continue;
 
    const intervals = runs.slice(1).map((r, i) => daysBetween(runs[i].end, r.start));
    const durations = runs.map(r => r.duration_days);
 
    const intervalMean = mean(intervals);
    const intervalStd = std(intervals);
    const durationMean = mean(durations);
    const durationStd = std(durations);
 
    if (intervalStd / intervalMean < 0.3) {
      loops.push({
        cluster_id: clusterId,
        occurrences: runs.length,
        avg_interval_days: intervalMean,
        avg_duration_days: durationMean,
        confidence: 1 - (intervalStd / intervalMean),
        runs: runs
      });
    }
  }
 
  return loops;
}
Surfacing loops to the user
When a new run of a known-looping cluster starts, surface this in the chat interface:
"You've entered your [user-confirmed cluster name] again. This is occurrence #5 in the past 6 months. Average duration: 12 days. Last 4 times this state was preceded by [trigger pattern]."
This is not a notification. It's a passive surface — only shows up if the user opens the chat. Don't email, don't push.

2.7 — Drift detection (stated vs lived)
Drift is the gap between what the user says they value and what they actually do. We have this data — content_type field on every memory (§2.2).
The algorithm
For each user, every Sunday, pull the last 30 days of memories.
Split into stated (content_type = 'stated') and lived (content_type = 'lived').
Generate an embedding-based summary of each side.
Ask Claude to surface the largest gaps.

2.8 — Entity correlations (who shifts you, what shifts you)
We extracted entities at ingestion time (§2.2). Now we compute correlations between entities and clusters.
The algorithm
For each entity that's been mentioned in ≥5 memories AND across ≥2 clusters, compute:
Co-occurrence rate — % of days where this entity is mentioned AND the user is in cluster X.
Compare to baseline — % of all days the user is in cluster X.
Lift = co-occurrence rate / baseline rate. Lift > 1.5 means meaningful positive correlation. Lift < 0.5 means meaningful negative correlation.
Code
function computeEntityCorrelations(userId: string) {
  const entities = getEntitiesWithMinMentions(userId, 5);
  const clusters = getConfirmedClusters(userId);
  const results = [];
 
  const totalDays = getDaysWithData(userId);
 
  for (const entity of entities) {
    const entityDays = getDaysWithEntity(userId, entity.id);
    
    for (const cluster of clusters) {
      const clusterDays = getDaysInCluster(userId, cluster.id);
      const intersection = entityDays.filter(d => clusterDays.includes(d));
      
      const cooccurrenceRate = intersection.length / entityDays.length;
      const baselineRate = clusterDays.length / totalDays.length;
      const lift = baselineRate > 0 ? cooccurrenceRate / baselineRate : 0;
      
      if (lift > 1.5 || lift < 0.5) {
        results.push({
          entity_id: entity.id,
          cluster_id: cluster.id,
          lift: lift,
          cooccurrence_count: intersection.length,
          direction: lift > 1.5 ? 'positive' : 'negative'
        });
      }
    }
  }
 
  return results;
}

2.9 — Forward inference (probabilistic next-state)
Given the user's current cluster, their cluster history, and any active loops, estimate likely trajectories for the next 14 days. Always cite. Never positioned as prediction.
V1 implementation — simple, not fancy
Build a transition matrix from the user's history. For each cluster pair (A → B), count how often the user transitioned from A to B. Normalize per row to get transition probabilities.
function buildTransitionMatrix(userId: string) {
  const assignments = getClusterAssignmentsOrdered(userId);  // ordered by date
  const transitions = {};
 
  for (let i = 0; i < assignments.length - 1; i++) {
    const from = assignments[i].cluster_id;
    const to = assignments[i+1].cluster_id;
    if (!from || !to) continue;
    if (from === to) continue;  // same cluster, not a transition
    transitions[from] = transitions[from] || {};
    transitions[from][to] = (transitions[from][to] || 0) + 1;
  }
 
  // Normalize
  const probs = {};
  for (const from in transitions) {
    const total = Object.values(transitions[from]).reduce((a, b) => a + b, 0);
    probs[from] = {};
    for (const to in transitions[from]) {
      probs[from][to] = transitions[from][to] / total;
    }
  }
 
  return probs;
}
Surfacing
Forward inference is on-demand only in V1. No auto-surface. User clicks "Where am I headed?" in the Mind Map tab. We respond:
"You're currently in [cluster name]. Based on your past behavior, the most likely transitions are: [next cluster A] 40%, [next cluster B] 30%, [back to current] 20%, [other] 10%. The last 4 times you exited this cluster, the trigger was [pattern]. Average time in current cluster from prior history: 8 days. You've been in it for 4."
TONE — Stay honest about uncertainty. Never say "you will" — always say "based on prior cycles, the probability is X." Always show the data. The product is a mirror, not an oracle.

Appendix B — Math reference
Plain-language explanations of the math we use. Read once. You don't need to be a statistician.
B.1 — UMAP (Uniform Manifold Approximation and Projection)
Takes high-dimensional data (our 1556-dim state vectors) and projects them into a lower-dimensional space (we use 8 dimensions) while preserving the structure of how points relate to each other.
Why we use it: HDBSCAN clusters better in lower dimensions. 1556 dimensions is too many for density-based clustering to work well — distance becomes meaningless at very high dimensions (the curse of dimensionality).
Key parameters:
n_components=8 — output dimensionality. We use 8 because empirically it's the sweet spot for clustering personal behavioral data.
n_neighbors=15 — how many nearby points to consider when building the structure. Larger = more global structure, smaller = more local structure. 15 is the default and works well.
random_state=42 — so we get reproducible results across runs. Important for debugging.
GOTCHA — UMAP is stochastic by default — different runs can give different outputs. Always set random_state. If you don't, you'll get different clusters every week and the user will be confused.
B.2 — HDBSCAN (Hierarchical Density-Based Spatial Clustering of Applications with Noise)
Density-based clustering algorithm. Finds groups of points that are densely packed together in space, and labels anything not in a dense group as noise (-1).
Why we use it (instead of K-Means): K-Means forces every point into a cluster, even outliers. It also requires you to specify K upfront. HDBSCAN admits "this point doesn't belong anywhere" as a valid output and figures out K on its own.
Key parameters:
min_cluster_size=5 — minimum number of points to call something a cluster. Lower = more clusters but each less stable. Higher = fewer but more robust.
min_samples=3 — how many neighbors a point needs to be considered a "core" point. Lower = more aggressive clustering, more noise. Higher = more conservative.
Tuning notes from our context:
For a user with 30 days of data, use min_cluster_size=4.
For a user with 90+ days of data, use min_cluster_size=5.
If a user gets only 1-2 clusters from a 90-day run, lower min_cluster_size to 3. They might be a very routine person.
If a user gets 12+ clusters, raise min_cluster_size to 7. They're likely fragmenting.
B.3 — Recurrence (the simple version we use)
Real recurrence quantification analysis (RQA) is complex. We use a simplified version:
Get the ordered sequence of cluster assignments for the user (day 1: cluster A, day 2: cluster A, day 3: cluster B, ...).
Find contiguous runs of each cluster: [{cluster: A, start: day 1, end: day 5}, {cluster: B, start: day 6, end: day 8}, ...].
For each cluster that has 3+ runs, compute the time gap between consecutive runs (intervals).
If the intervals have low variance (stddev < 30% of mean), the cluster is recurring with stable periodicity — call it a loop.
Why this works: it's the simplest possible test for "is this happening at a consistent rhythm?" It will miss some real patterns (e.g., loops with varying intervals but consistent durations) and catch some false positives (random clustering of a few intervals), but for V1 it's good enough.
B.4 — Lift (for entity correlations)
Statistical measure of how strongly two events are associated. Computed as:
lift = P(in cluster X | entity Y mentioned) / P(in cluster X)
Interpretation:
Lift = 1.0: entity and cluster are independent (no association).
Lift > 1.5: meaningful positive association. When this entity shows up, the user is more likely to be in this cluster than baseline.
Lift < 0.5: meaningful negative association. When this entity shows up, the user is less likely to be in this cluster than baseline.
Between 0.5 and 1.5: we don't surface — too weak to be reliable.

D.4 — Common pitfalls
Forgetting RLS. Supabase has Row-Level Security. Every new table needs RLS policies that scope reads/writes to the authenticated user_id. Don't forget.
Forgetting indexes. Every new table that's queried by user_id + date needs an index on (user_id, date). Slow queries will tank the user experience.
Not handling Claude rate limits. If Claude returns 429, wait with exponential backoff. Don't crash the cron job.
Hardcoding model names. Put model names in a single config file. We'll swap models for cost optimization later.
Synchronous Claude calls in webhook handlers. Webhooks must return 200 fast. Push the Claude call into a background queue (or use waitUntil() in Vercel).
